Code Review 是 Anthropic 官方做的一个 Claude Code 插件，作用很直接：把 pull request 的代码审查流程自动化。

它不是那种泛泛地“帮你看看代码有没有问题”的插件，而是一个相对工程化的 reviewer pipeline：会看 PR 是否值得审查，会找相关 CLAUDE.md 规范，会总结变更，然后拉起多路 reviewer agents 并行审查，最后再对每条 issue 做一轮 confidence scoring，只把高置信度的问题发回 GitHub。

截至 2026-04-03，我打开官方插件页 claude.com/plugins/code-review 时，它显示：

  • Anthropic Verified
  • Made by Anthropic
  • Installs: 191,903

而在我同一天看到的 claude.com/plugins 浏览列表里，它排在前列，位于 Frontend Design、Superpowers、Context7 之后。

它是什么

官方插件页：Code Review

这个插件的定位是：

  • 面向 pull request；
  • 使用多个 specialized agents 并行审查；
  • 对每个发现的问题做 confidence-based filtering；
  • 尽量减少 false positives；
  • 最终把 actionable feedback 直接发到 GitHub。

说白一点，它不是在替代人类 reviewer，而是在做一轮“高置信度预审”。

这个思路其实很对，因为 PR review 最大的痛点往往不是“没东西查”，而是：

  • 误报太多；
  • reviewer comment 太吵；
  • 小毛病太多，真正重要的问题反而被淹掉；
  • 项目规范分散在 CLAUDE.md、代码注释、历史上下文里，普通 lint 根本抓不到。

Code Review 插件解决的就是这类问题。

最基本的用法：

它适合这些场景：

  • 非 trivial 的 PR；
  • 涉及关键路径的改动；
  • 多人协作仓库；
  • 项目里已经有比较明确的 CLAUDE.md 规范。

不太适合的场景：

  • draft PR；
  • 已关闭 PR；
  • 很小的自动化 PR；
  • 急着马上 merge 的紧急 hotfix。

而且这些场景里，有不少它会自动跳过。

它到底会做什么

公开 README：

  • 插件 README：plugins/code-review/README.md
  • 命令实现：plugins/code-review/commands/code-review.md

README 给的总结是：

  1. 检查这个 PR 是否值得 review；
  2. 找相关的 CLAUDE.md 指南文件；
  3. 总结 PR 改了什么；
  4. 启动多路 agent 并行审查；
  5. 对每个 issue 做 0-100 的 confidence scoring；
  6. 把低于阈值的 issue 过滤掉；
  7. 最后把高置信度问题评论回 GitHub。

审查 agent 怎么分工

从 commands/code-review.md 看，真正的分工其实更细。

它会先用 Haiku agent 做几件准备工作：

  • 判断这个 PR 是不是该 review；
  • 找相关 CLAUDE.md；
  • 给 PR 做一个 change summary。

然后再拉起 5 个并行 Sonnet agents 去看不同维度：

  1. 检查是否违反 CLAUDE.md
  2. 只盯变更本身找 obvious bugs
  3. 看 git blame / history 找上下文问题
  4. 看以前相关 PR 的评论里有没有相同模式
  5. 看被修改文件里的代码注释有没有明确约束

这套拆法非常实用，因为它避免让一个 reviewer 一次背太多任务。

它怎么减少误报

这插件最重要的一层，不是多 agent，而是后面的 confidence scoring。

命令实现里写得很死：

  • 每个 issue 再交给一个 Haiku agent 去打分；
  • 分数是 0 到 100；
  • 默认只保留 80 以上 的问题；
  • 低于 80 的，直接过滤掉，不发回去。

这个机制的价值非常大。

因为普通 AI code review 工具最常见的问题就是：

  • 会说得头头是道；
  • 但里面很多问题其实只是“可能有问题”；
  • 最后让人类 reviewer 疲劳。

Code Review 这个插件就是在专门抑制这种噪音。

它的工作原理，能看懂吗

可以，而且这次比 Frontend Design 更“工程化”，也更像一个真正的自动化 workflow。

第一层：它是一个命令型插件

plugin.json 很短：

但真正的逻辑在 commands/code-review.md 里。

也就是说，它不是单靠一个 skill 影响模型习惯，而是提供了一个结构化命令：/code-review。

第二层：核心是一个严格编排过的审查脚本

如果你去看 commands/code-review.md，会发现它不是“随便 review 一下”，而是把步骤写得非常死：

  • 先 eligibility check；
  • 再找 CLAUDE.md；
  • 再总结 PR；
  • 再 5 路并行 review；
  • 再 issue 逐条打分；
  • 再按阈值过滤；
  • 再二次 eligibility check；
  • 最后用 gh pr comment 发评论。

这其实已经不是一个简单 prompt，而更像一个审查流程 DSL。

第三层：它依赖 GitHub CLI，而不是 Web Fetch

命令 frontmatter 里明确允许的工具是 gh 相关 Bash 命令，比如：

  • gh pr view
  • gh pr diff
  • gh pr list
  • gh pr comment

而命令正文里还特别强调：

用 gh 和 GitHub 交互，不要用 web fetch。

这说明它的设计前提就是：

  • 运行环境在本地开发机 / CI / Claude Code 终端；
  • 当前仓库已经有 GitHub remote；
  • gh 已经登录好。

这也解释了为什么它很适合真正的 PR review workflow，而不是纯静态代码分析。

第四层：它不是追求“找到更多问题”，而是追求“只发值得发的问题”

我觉得这是这个插件最值得学的地方。

从实现里看，它明确把这些都当成 false positives 或无需发回的问题：

  • pre-existing issues
  • lint/typecheck/compiler 会抓到的问题
  • pedantic nitpicks
  • 没改到的代码上的历史问题
  • 很可能是 intentional behavior change 的东西

这其实是一种很成熟的 code review 价值观：

不要因为“能说”就都说出来，review comment 是一种稀缺注意力资源。

一个典型使用场景

假设你提了一个包含认证改动的 PR。

你在 PR branch 上运行：

然后它大致会这样做：

  1. 看这个 PR 是不是 draft / closed / 已经 review 过；
  2. 找当前仓库里根目录和改动目录下的 CLAUDE.md；
  3. 总结这个 PR 改了哪些文件、做了什么；
  4. 并行派 5 个 reviewer 去看；
  5. 再对每个 issue 做独立置信度打分；
  6. 只留下 >=80 的问题；
  7. 把结果评论回 GitHub。

最后 GitHub comment 会包含：

  • 简短的问题描述；
  • 问题为什么成立；
  • 对应 CLAUDE.md 或 bug 依据；
  • 直接跳到具体代码行的 GitHub 链接。

适合谁

我觉得它特别适合：

  • GitHub PR 流程已经比较规范的团队；
  • 项目里本来就维护 CLAUDE.md 规范；
  • 想用 AI 做第一轮 reviewer，但很怕噪音；
  • 想把“规则审查 + obvious bug 审查 + 历史上下文审查”组合起来。

如果你是：

  • 单人项目；
  • 不怎么走 PR；
  • 没有 GitHub CLI；
  • 不维护 CLAUDE.md；

那这个插件的价值会下降很多。

我的判断

如果只看公开实现，我会把 Code Review 归类成一个非常“实战派”的官方插件：

  • 它不是靠一段审美 prompt 起作用；
  • 也不是靠 hooks 改会话默认行为；
  • 它本质上是一个 PR review orchestration command；
  • 多 agent 并行只是第一层；
  • 真正让它有用的是后面的 confidence scoring 和噪音过滤。

也就是说，它最核心的价值不是“更聪明”，而是“更克制”。

而大多数 AI code review 工具，恰恰缺的就是这种克制。
