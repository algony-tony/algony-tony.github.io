Code Simplifier 是 Anthropic 官方做的一个 Claude Code 插件，目标也非常直接：在不改变功能的前提下，把刚刚改过的代码再收拾一遍，让它更清晰、更一致、更容易维护。

它不是做 bug 修复主流程的，也不是做 PR review 编排的。它更像一个“收尾型 agent”：当你已经写完一段代码，或者刚修改完几个函数之后，它会从可读性、复杂度、命名、一致性这些维度出发，再做一次 refinement。

截至 2026-04-03，我打开官方插件页 claude.com/plugins/code-simplifier 时，它显示：

  • Anthropic Verified
  • Made by Anthropic
  • Installs: 159,908

同一天我在 claude.com/plugins 的浏览列表里看到，它也排在前排，位于 Code Review 之后。

它是什么

官方插件页：Code Simplifier

这个插件页给它的定义很短，但非常清楚：

  • simplifies and refines recently modified code
  • preserving functionality and consistency

这两句其实已经把边界说完了：

  1. 只动最近改过的代码，默认不扫全仓；
  2. 不改变行为，只是优化实现方式。

所以它解决的不是“代码不能跑”，而是另一类很常见的问题：

代码能跑，但写得拧巴，或者刚好跨过了“以后会越来越难读”的那条线。

比如：

  • 嵌套层级太深；
  • 多个条件写成了可读性很差的三元表达式；
  • 变量名很随手；
  • 有点重复，但还没严重到值得单开一个 refactor task；
  • 风格和项目已有代码不一致。

Code Simplifier 做的，就是把这些“小别扭”在代码刚出炉的时候顺手处理掉。

怎么用

官方插件页：Code Simplifier

这个插件的使用方式和 Code Review 不太一样。

它没有一个公开可见、非常明确的 /something 命令入口；官方插件页写的是：

  • 它会 proactively 地作用在 recently modified code 上；
  • 你也可以直接要求它看特定文件或代码段；
  • 示例 prompt 类似：
    • “simplify this component”
    • “refine the recently changed functions for clarity”

所以它更像一个 agent / subagent，而不是一个大工作流命令。

适合什么时候用

这个插件特别适合：

  • 你刚写完一段功能；
  • 你知道代码能跑，但还不够顺；
  • 你想做一次小型 clean-up；
  • 你不想开一个很大的 refactor 分支；
  • 你想让 Claude 按项目风格收一下边角。

它不太适合：

  • 你还没把功能写对；
  • 你现在最需要的是 debug；
  • 你要重构的是整个模块；
  • 你想重新做架构，不只是简化实现。

它的工作原理，能看懂吗

可以，而且它比 Code Review 还要简单。

第一层：它本质上是一个 agent prompt

plugin.json 很短：

而真正有内容的，是：

  • agents/code-simplifier.md

也就是说，从公开实现看，它不是一个复杂的 workflow plugin，更像一个专职 agent。

第二层：它强调“行为不变”

这个 agent prompt 里第一条就是：

  • Never change what the code does - only how it does it.

这条其实就是整个插件的地基。

因为 code simplification 最大的风险就在这里：

你以为你是在让代码更优雅，结果其实把逻辑改了。

这个插件一上来就把边界钉死了：

  • 原始行为不能变；
  • feature / output / behavior 都要保留；
  • 它只能动实现方式。

第三层：它不是追求“更短”，而是追求“更清楚”

这是我觉得它最好的地方。

很多“简化代码”的工具，最后都容易滑向一个坑：

  • 行数更少；
  • 表达更紧；
  • 但也更难读。

而 agents/code-simplifier.md 里专门强调了：

  • readable, explicit code over overly compact solutions
  • avoid nested ternary operators
  • choose clarity over brevity
  • 不要为了 fewer lines 牺牲可读性

也就是说，它不是在做 code golf，而是在做 maintainability-oriented cleanup。

第四层：它默认只盯最近改动

prompt 里还写得很清楚：

  • 默认只 refine recently modified code
  • 除非用户明确要求，不扩大 scope

这一点很关键，因为它把这个插件控制在一个很好用的范围里：

  • 不会动不该动的老代码；
  • 不会一上来就变成全仓风格战；
  • 更适合放在“刚改完一轮代码之后”。

第五层：它会参考项目标准

agent prompt 里还要求它遵守项目里已有的 CLAUDE.md 规范，比如：

  • import 排序
  • ES modules
  • function declaration 风格
  • top-level return type
  • React props pattern
  • naming conventions

所以它并不是在凭空输出“它觉得更漂亮的代码”，而是：

先读项目标准，再在标准内做简化。

这让它比纯粹的格式化器更聪明，又比完全自由发挥的 AI 更克制。

它会具体改什么

从公开 agent prompt 看，它主要会盯这些点：

  • 减少不必要的复杂度和 nesting
  • 消除冗余代码和多余抽象
  • 改善变量和函数命名
  • 合并相关逻辑
  • 移除解释显而易见代码的无用注释
  • 避免 nested ternary
  • 保持调试和扩展友好

但它也明确写了不要过度简化，比如不能：

  • 把多个关注点硬塞进一个函数；
  • 为了短而写得太 clever；
  • 去掉本来有帮助的 abstraction；
  • 让代码更难 debug。

所以这个插件整体非常像一个有经验的 senior engineer 在做这件事，而不是一个“压缩代码字节数”的工具。

一个典型使用场景

比如你刚让 Claude 帮你改完一个 React 组件。

功能已经对了，但你看了一眼，发现：

  • 条件判断有点乱；
  • props 处理不够统一；
  • 有几个变量名很随意；
  • 某个 render 分支用了嵌套三元表达式。

这时候你不一定想重做整个组件，也不一定想单开一个重构任务。你只需要说：

这个时候它最有价值。

适合谁

我觉得它最适合：

  • 已经让 Claude 帮你完成了功能实现的人；
  • 你认可“先做对，再做顺”这个节奏；
  • 你不想每次都手工写一大堆“请顺手再优化代码风格”；
  • 你项目里本来就有比较明确的代码规范。

如果你想要的是：

  • 自动修 bug；
  • 自动做大型架构重构；
  • 自动做 PR comment；

那这不是它的方向。

我的判断

如果只看公开实现，Code Simplifier 是一个非常典型的“后处理型官方 agent”：

  • 结构很轻；
  • 没有复杂 orchestration；
  • 主要靠一个 agent prompt 起作用；
  • 但 prompt 的边界设得很好；
  • 核心价值在于“清晰优先、行为不变、范围克制”。

也就是说，它并不是那种惊艳型插件，但它很容易成为日常 workflow 里一个很实用的小工具。

它适合放在这样的位置：

写完一轮代码之后，在 commit 之前，再让它做一轮维护性打磨。
