OpenAI 的 Codex 现在已经不只是一个“会补全代码的模型名”，而是一整套 coding agent 产品：有 Web、桌面 App、IDE 扩展，也有最适合程序员折腾的 Codex CLI。

这篇主要讲终端里的 Codex，也就是仓库 openai/codex 对应的 CLI。它的定位很直接：让 agent 在你的本地仓库里读文件、改文件、跑命令、开子代理，并把整个过程放在终端里完成。

截至 2026-04-04，我打开 GitHub 仓库页 openai/codex 时看到：

  • 仓库 star 大约 73k；
  • 最新 release 是 0.118.0；
  • 官方 README 里的描述是：“Lightweight coding agent that runs in your terminal”。

认证

官方 README：openai/codex

Codex CLI 现在最推荐的用法不是一上来就配 API Key，而是直接用 ChatGPT 账号登录。

README 里写得很清楚：运行 codex 之后，可以选择 Sign in with ChatGPT。官方建议把它作为 ChatGPT Plus、Pro、Team、Edu、Enterprise 计划的一部分来用。

当然，也还是可以走 API Key 模式，只是这个需要额外配置。

最基本的用法

官方文档：Codex CLI features

Codex CLI 最核心的工作方式是 交互式全屏终端 UI。

官方文档里对 interactive mode 的总结我觉得很准：

  • 它可以直接读取仓库；
  • 可以修改文件；
  • 可以执行命令；
  • 你能在终端里实时看到它先解释计划，再执行动作；
  • 每一步都可以在会话中直接批准或拒绝。

也就是说，它不是“把网页聊天窗口搬到终端里”，而是真正的本地 agent。

会话恢复

官方文档：Codex CLI features

Codex 会把本地会话存下来，所以你不用每次都重新解释项目背景。

文档里还提到，非交互式 codex exec 运行也支持 resume，所以它并不是“开一次用一次”的一次性工具。

常用斜杠命令

官方文档：Slash commands in Codex CLI

Codex CLI 的斜杠命令不算特别花哨，但很实用，基本都是“在当前会话里动态调整 agent 行为”。下面挑几个日常最常用的。

命令 说明
/permissions 调整当前会话里哪些动作可以不经询问直接执行
/agent 查看或切换当前 agent thread
/apps 浏览 connector / app，并把它插入当前提示词
/clear 清空当前对话，重新开始
/compact 压缩对话上下文，节省 token
/copy 复制最近一次完成的输出
/diff 查看当前改动 diff，包括未跟踪文件
/init 在当前目录生成一个 AGENTS.md 模板
/mcp 查看已配置的 MCP 工具
/mention 把文件或目录显式附到当前对话里
/model 切换模型和 reasoning effort
/fast 给 GPT-5.4 切换 Fast mode
/plan 切到只读 planning 模式先出方案
/personality 改 Codex 的回答风格
/ps 查看后台终端及最近输出
/resume 恢复历史会话
/new 新开一段对话
/quit 退出

这一套命令的思路非常明显：Codex 把“模型本身”和“会话控制”分开了。

模型负责做事，slash commands 负责调度当前工作环境。

模型

官方文档：Codex CLI features

截至 2026-04-04，我在官方文档里看到，Codex 里日常推荐模型是 gpt-5.4。

文档里的原话大意是：对于大多数任务，gpt-5.4 是推荐模型；它结合了更强推理能力、原生 computer use，以及更广的专业工作流能力。对特别追求速度的任务，ChatGPT Pro 用户还能用 GPT-5.3-Codex-Spark 的 research preview。

所以现在的 Codex 其实不是“只能绑死一个 coding model”，而是：

  • 主力模型用 gpt-5.4；
  • 有更快的 spark 型选择；
  • 在 CLI 里可以用 /model 动态切换。

核心概念

AGENTS.md

官方文档导航里把它单独列成了一项配置能力；CLI 里也可以用 /init 直接生成。

这玩意可以粗暴理解成：给 Codex 的项目内长期提示词。

也就是你希望它每次进入这个仓库时都知道的东西，比如：

  • 测试命令；
  • commit 规范；
  • 哪些目录不能动；
  • 代码风格；
  • 团队约定；
  • PR 提交流程。

这点和 CLAUDE.md、GEMINI.md 的思路是同一类：把“口头交代”变成“项目内常驻上下文”。

Subagents

官方文档：Subagents

Codex 现在已经支持相当完整的 subagent 工作流。

文档里给的核心点有几个：

  • Codex 可以显式按你的要求并行拉起多个 subagent；
  • 这些 subagent 各自做自己的工具调用和推理，再把结果汇总回来；
  • 默认内置了 default、worker、explorer 三类 agent；
  • 自定义 agent 可以放在 ~/.codex/agents/ 或项目内 .codex/agents/；
  • /agent 可以查看和切换 agent thread。

这个能力很关键，因为它让 Codex 不再只是“一个大模型串行做所有事”，而是开始像一个小型 orchestrator。

MCP / Apps / Skills / Plugins

从官方文档导航看，Codex 现在把这几个层面都已经补齐了：

  • MCP：连接 GitHub、Slack、数据库之类外部系统；
  • Apps / Connectors：更偏“把外部能力插到当前任务里”；
  • Skills：可复用工作流提示；
  • Plugins：更像一个完整扩展包；
  • Hooks：把某些动作固化成一定会执行的自动步骤。

所以 Codex CLI 的真实定位，不是“终端里的聊天工具”，而是 OpenAI 自己的本地 agent runtime。

它和 Claude Code 的差别，最值得看的是什么

我觉得不是表面上的“命令长得像不像”，而是默认设计取向。

Codex 现在给我的感觉是：

  • 把 AGENTS.md 放得很核心；
  • 把 subagents 做成正式能力，而不是隐藏技巧；
  • 把 slash commands 当作运行时控制面板；
  • 把 CLI / IDE / App / Web 做成一个统一的产品族。

所以如果你已经习惯 agent 编程，那 Codex 值得看的地方，不只是“它能不能改代码”，而是它把 “项目规则、运行时安全、子代理协作、会话恢复” 这些东西做得越来越像一个正式开发环境。
