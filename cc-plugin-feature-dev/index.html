Feature Dev 是 Anthropic 官方做的一个 Claude Code 插件，它的目标不是帮你写某一段具体代码，而是：把“开发一个新功能”这件事情，流程化成一个带阶段、带 agent 分工、带审查和确认点的工作流。

如果说 Frontend Design 是在纠正审美默认值，Code Review 是在自动化 PR 审查，Code Simplifier 是在做代码收尾打磨，那么 Feature Dev 就是在解决另一个很典型的问题：

Claude 一接到“做个新功能”这种任务，太容易直接冲进代码里。

截至 2026-04-03，我打开官方插件页 claude.com/plugins/feature-dev 时，它显示：

  • Anthropic Verified
  • Made by Anthropic
  • Installs: 143,911

它是什么

官方插件页：Feature Dev

插件页对它的总结很准确：

  • 它是一个 structured 7-phase workflow；
  • 不鼓励直接开始写代码；
  • 会先理解代码库；
  • 会澄清需求；
  • 会做架构设计；
  • 会在实现前后做 quality review。

这其实是一种很“软件工程流程化”的插件。

它不追求最短路径，而是追求：

  • 先理解上下文；
  • 再消除歧义；
  • 再设计实现方案；
  • 再实施；
  • 再复查质量。

所以它特别适合“新增功能”这种任务，而不太适合那种一眼能做完的小改动。

它的 7 个阶段

公开 README：

  • 插件 README：plugins/feature-dev/README.md
  • 命令实现：plugins/feature-dev/commands/feature-dev.md

这个插件最核心的价值，就在这 7 个阶段里。

Phase 1: Discovery

先搞清楚你到底要做什么。

如果需求本身是模糊的，它会先问：

  • 你要解决什么问题；
  • 这个 feature 到底应该做什么；
  • 有没有约束或额外要求。

Phase 2: Codebase Exploration

这是它和一般 agent 最大的差别之一。

它不会马上改代码，而是先派 2~3 个 code-explorer agents 并行探索：

  • 有没有类似功能；
  • 当前架构怎么分层；
  • 有没有相关抽象和 extension points；
  • 现有代码是怎么流转的。

而且 explorer agents 不只是扫一眼文件名，它们会 trace execution path、map architecture layers、列出关键文件清单。

Phase 3: Clarifying Questions

这是整个插件里我觉得最重要的一步之一。

它会在理解代码库以后，把还不明确的地方一次性列出来问你，比如：

  • edge cases
  • error handling
  • integration points
  • backward compatibility
  • performance expectations

重点是：它在开始设计前，先尽量把歧义收敛掉。

Phase 4: Architecture Design

这个阶段它会再派 2~3 个 code-architect agents，从不同视角给出实现思路：

  • 最小改动
  • 更干净的架构
  • pragmatic balance

然后对比 trade-off，再向你给 recommendation。

Phase 5: Implementation

这里才开始真正写代码。

而且命令实现里写得非常明确：

没有用户批准，不要开始实现。

也就是说，它不只是探索和设计，还把“什么时候允许动手”也做成了 gate。

Phase 6: Quality Review

实现完以后，它还会再派 3 个 code-reviewer agents 并行从不同角度看：

  • simplicity / DRY / elegance
  • bugs / correctness
  • conventions / abstractions

然后把结果汇总给你，再问你：

  • 现在修
  • 以后修
  • 还是就这样继续

Phase 7: Summary

最后收尾：

  • 标记 todo 完成；
  • 总结做了什么；
  • 记录关键决策；
  • 列出改动文件；
  • 给出 next steps。

它的工作原理，能看懂吗

可以，而且这是三个插件里最容易看出“工程编排感”的一个。

第一层：它本质上是一个 workflow command

plugin.json 只有元数据，真正有内容的是：

  • commands/feature-dev.md

这意味着它的核心不是单个 skill，也不是单个 reviewer agent，而是一个 orchestrated command：/feature-dev。

第二层：它把“做功能”拆成阶段机

如果你看 commands/feature-dev.md，会发现它写得像一份流程手册，不像普通 prompt：

  • 每个阶段有 Goal；
  • 每个阶段有 Actions；
  • 哪一步要 wait for user answers；
  • 哪一步不要在没批准前实现；
  • 哪一步要并行派 agent。

也就是说，它其实是在把 Claude Code 变成一个按 phase 跑的 feature PM + architect + implementer + reviewer 组合体。

第三层：核心 agent 分工很清楚

Feature Dev 公开出来的 3 个 agent，各自职责很明确：

code-explorer

作用是深挖现有代码库：

  • 找 entry points；
  • trace code flow；
  • map architecture layers；
  • 找 dependencies 和 side effects；
  • 列出必须阅读的关键文件。

它本质上是在替 Claude 做“先摸清地形”。

code-architect

作用是基于现有模式做架构设计：

  • 分析现有 conventions；
  • 做 architecture decision；
  • 明确组件设计；
  • 明确 implementation map；
  • 给 phased build sequence。

它其实是在做“把探索结果转成设计蓝图”。

code-reviewer

作用是实现后看质量：

  • 看项目规范；
  • 看 bug；
  • 看代码质量；
  • 只报高置信度问题。

所以这三个 agent 组合起来，基本就是：

先理解，再设计，再复查。

第四层：它最核心的价值其实不是“更会写代码”，而是“更晚开始写代码”

我觉得这点特别关键。

普通 Claude Code 最常见的问题，是它太容易在信息不全的时候就开始动手。

Feature Dev 用 7-phase workflow 在强行压制这种冲动：

  • 需求不清？先问。
  • 代码库没懂？先探索。
  • 设计没定？先比方案。
  • 用户没批准？先别实现。

这让它特别适合复杂 feature，而不是简单 patch。

一个典型使用场景

假设你想在一个已有系统里新增 OAuth 登录。

你运行：

然后它可能会这样走：

  1. 先确认你要支持哪些 provider；
  2. 派 explorer agents 去看现有 auth、session、security middleware；
  3. 汇总后问你：
    • 要不要和现有 auth 共存？
    • token 存什么？
    • error handling 怎么做？
  4. 再派 architect agents 给出几种方案；
  5. 让你选方案；
  6. 你批准后开始实现；
  7. 实现后再派 reviewer agents 查质量；
  8. 最后给你 summary。

这个过程比直接“帮我加 OAuth”慢，但出错概率会低很多。

适合谁

这个插件特别适合：

  • 要做的 feature 比较大；
  • 改动会涉及多个文件或模块；
  • 你不想让 Claude 一上来就乱改；
  • 你希望它先做 codebase study；
  • 你认可“先设计后实现”的节奏。

它不太适合：

  • 单行小修；
  • 简单 bug fix；
  • 你已经把要求说得极度清楚、只想赶紧改完；
  • 你不想被问 clarifying questions。

我的判断

如果只看公开实现，Feature Dev 是一个非常典型的 Anthropic 官方 workflow 插件：

  • 它不是单纯 prompt；
  • 也不是一个单点技能；
  • 它是一个 command-driven orchestration；
  • 通过 explorer / architect / reviewer 三类 agents，把 feature development 这件事拆阶段执行；
  • 最核心的价值是：延迟实现，先理解、先澄清、先设计。

所以如果你问我它最适合什么任务，我会说：

“不是不会写，而是容易写太快”的那类功能开发任务。

对于复杂功能，它很有价值； 对于简单任务，它会显得有点重。
