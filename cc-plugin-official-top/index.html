Anthropic 在 claude.com/plugins 提供了官方插件目录。截至 2026-04-03，在 Browse plugins 的 Claude Code 列表里，安装量前几名的官方插件依次是：

1 Frontend Design 413,623
2 Superpowers 294,839
3 Context7 212,236

其中 Superpowers 和 Context7 不是 Anthropic 自己出的。这篇介绍 Anthropic 官方做的四个：Frontend Design、Code Review、Code Simplifier、Feature Dev。

Frontend Design

官方插件页：claude.com/plugins/frontend-design

安装量 413,623，排名第一。

这个插件的目标非常明确：让 Claude 写出来的前端，不要一眼看上去就是”标准 AI 味”的页面。

它不负责接数据库、跑浏览器自动化，也不负责项目管理。它做的事情很集中：当你让 Claude 生成页面、组件、landing page、dashboard 这类界面时，把 Claude 往”更有设计感、更敢做风格选择”的方向推一把。

怎么用

基本不需要记命令。插件会在你做 frontend work 的时候自动触发，直接提需求就行：

想让效果更好，最好给这几类信息：

信息 例子
产品类型 AI 安全公司、音乐 App、设计工具、消费品牌
目标用户 开发者、设计师、B 端管理者、年轻消费者
技术约束 React / Vue / Tailwind / 纯 HTML
风格方向 brutalist、editorial、luxury、playful、retro-futuristic
重点区域 hero、定价区、数据表、导航、卡片、动效

工作原理

从公开实现来看，它是一个非常轻的 skill 型插件：没有 MCP server、没有 hooks、没有 commands，主要靠 skills/frontend-design/SKILL.md 起作用。

skill 的 description 写得非常明确——”Use this skill when the user asks to build web components, pages, or applications”——Claude Code 根据 description 匹配自动触发。

SKILL.md 里做了几件事：

先定设计方向，再写代码。 它要求 Claude 在编码前先确认四件事：Purpose、Tone、Constraints、Differentiation，避免模型直接退回安全模板。

强制做有风格的选择。 skill 要求 Claude 必须选定一个明确的审美方向，给出了很多极端风格供参考：brutally minimal、maximalist chaos、retro-futuristic、luxury/refined、editorial/magazine 等，并明确说：

不要做一个平均值页面，要先选一个风格立场。

明确列出不要做什么。 这是最有效的部分：直接点名禁止一些高频 AI 默认项——Inter、Roboto、Arial 字体、紫色渐变白底配色、predictable layouts、cookie-cutter components。

它不是给 Claude 新增”设计能力”，而是通过 skill prompt 改变 Claude 在做前端时的默认审美偏好。

Anthropic 有一个配套 cookbook：Frontend Aesthetics: A Prompting Guide，精神和 SKILL.md 高度一致，可以理解为这个插件就是把那套 cookbook 产品化了。

适合谁

适合：经常让 Claude 出 landing page、做 demo 页、嫌 Claude 默认做出来的界面”太像 AI 做的”。

不太适合：内部管理后台、标准 B 端表单、设计系统严格约束下的企业页面——那类场景要”稳”，不需要风格冒险。

Code Review

官方插件页：claude.com/plugins/code-review

安装量 191,903，排名第四。

这个插件的作用很直接：把 pull request 的代码审查流程自动化。它不是”帮你看看代码有没有问题”，而是一个工程化的 reviewer pipeline：会判断 PR 是否值得审查，会找相关 CLAUDE.md 规范，会总结变更，然后拉起多路 reviewer agents 并行审查，最后对每条 issue 做 confidence scoring，只把高置信度的问题发回 GitHub。

怎么用

主入口是：

在 PR branch 上运行，它就会启动完整的自动审查流程。

适合：非 trivial 的 PR、涉及关键路径的改动、多人协作仓库、项目里有明确 CLAUDE.md 规范的场景。

不太适合：draft PR、已关闭 PR、很小的自动化 PR、急着马上 merge 的紧急 hotfix——很多这类情况它会自动跳过。

工作原理

审查分工非常细。 它先用 Haiku agent 做三件准备工作：判断 PR 是否该 review、找相关 CLAUDE.md、给 PR 做 change summary。然后再拉起 5 个并行 Sonnet agents 审查不同维度：

  1. 检查是否违反 CLAUDE.md
  2. 只盯变更本身找 obvious bugs
  3. 看 git blame / history 找上下文问题
  4. 看以前相关 PR 的评论里有没有相同模式
  5. 看被修改文件里的代码注释有没有明确约束

这套拆法避免了让单个 reviewer 一次背太多任务。

最重要的一层是 confidence scoring。 每个 issue 再交给一个 Haiku agent 打分，0 到 100，默认只保留 80 以上 的问题，低于阈值的直接过滤。

这个机制非常关键。普通 AI code review 工具最常见的问题，就是会说得头头是道，但很多”问题”其实只是”可能有问题”，最后让人类 reviewer 疲劳。Code Review 专门在抑制这种噪音。

它依赖 GitHub CLI，不用 web fetch。 命令里明确指定允许的工具是 gh 相关 Bash 命令（gh pr view、gh pr diff、gh pr comment 等），运行前提是本地已登录 gh，当前仓库有 GitHub remote。

从实现里看，它明确把这些当成 false positives 不发出去：pre-existing issues、lint/typecheck 会抓到的问题、pedantic nitpicks、没改到的代码上的历史问题、很可能是 intentional 的行为变更。这体现了一种成熟的 code review 价值观：

不要因为”能说”就都说出来，review comment 是一种稀缺注意力资源。

适合谁

适合：GitHub PR 流程规范的团队、项目里本来就维护 CLAUDE.md 规范、想用 AI 做第一轮 reviewer 但很怕噪音。

不太适合：单人项目、不走 PR、没有 GitHub CLI、不维护 CLAUDE.md。

Code Simplifier

官方插件页：claude.com/plugins/code-simplifier

安装量 159,908，排名第五。

这个插件的目标：在不改变功能的前提下，把刚刚改过的代码再收拾一遍，让它更清晰、更一致、更容易维护。

它不是做 bug 修复主流程的，也不是做 PR review 编排的。它更像一个”收尾型 agent”：当你已经写完一段代码，或者刚修改完几个函数之后，它会从可读性、复杂度、命名、一致性这些维度出发，再做一次 refinement。

它解决的不是”代码不能跑”，而是另一类很常见的问题：

代码能跑，但写得拧巴，或者刚好跨过了”以后会越来越难读”的那条线。

怎么用

它没有一个固定命令入口，官方插件页写的是：它会 proactively 地作用在 recently modified code 上，也可以直接要求它看特定文件或代码段。

适合：你刚写完一段功能、代码能跑但还不够顺、想做一次小型 clean-up、不想开一个大重构分支。

不太适合：功能还没写对、现在最需要 debug、要重构整个模块或重新做架构。

工作原理

从公开实现看，它的结构非常轻——主要就是 agents/code-simplifier.md 一个 agent prompt。

行为不变是第一条。 agent prompt 里开门见山：

Never change what the code does - only how it does it.

这条是整个插件的地基，因为 code simplification 最大的风险就是：你以为在让代码更优雅，结果把逻辑改了。

追求的是”更清楚”，不是”更短”。 很多简化代码的工具最终都滑向：行数更少、表达更紧、但也更难读。这个 agent prompt 里专门强调：

  • readable, explicit code over overly compact solutions
  • avoid nested ternary operators
  • choose clarity over brevity
  • 不要为了 fewer lines 牺牲可读性

它不是在做 code golf，而是在做 maintainability-oriented cleanup。

默认只盯最近改动。 prompt 里写明：默认只 refine recently modified code，除非用户明确要求，不扩大 scope。这让它不会动不该动的老代码，不会变成全仓风格战。

会参考项目标准。 它要求遵守项目里已有的 CLAUDE.md 规范，比如 import 排序、ES modules、function declaration 风格、naming conventions 等。先读项目标准，再在标准内做简化。

它主要会盯：减少不必要的复杂度和 nesting、消除冗余代码、改善变量和函数命名、合并相关逻辑、移除解释显而易见代码的无用注释。同时明确不过度简化——不能把多个关注点硬塞进一个函数，不能去掉本来有帮助的 abstraction。

适合谁

适合：已经让 Claude 帮你完成功能实现、认可”先做对，再做顺”这个节奏、不想每次都手工写一大堆”请顺手再优化代码风格”的人。

它最适合放在这个位置：写完一轮代码之后，在 commit 之前，再让它做一轮维护性打磨。

Feature Dev

官方插件页：claude.com/plugins/feature-dev

安装量 143,911。

这个插件的目标不是帮你写某一段具体代码，而是：把”开发一个新功能”这件事，流程化成一个带阶段、带 agent 分工、带审查和确认点的工作流。

它要解决的是一个很典型的问题：

Claude 一接到”做个新功能”这种任务，太容易直接冲进代码里。

7 个阶段

这个插件最核心的价值就在这 7 个阶段里。

Phase 1: Discovery。 先搞清楚你到底要做什么。需求模糊时，先问你要解决什么问题、这个 feature 应该做什么、有没有约束。

Phase 2: Codebase Exploration。 不马上改代码，而是先派 2~3 个 code-explorer agents 并行探索：有没有类似功能、当前架构怎么分层、有没有相关抽象和 extension points、现有代码是怎么流转的。explorer agents 不只是扫文件名，它们会 trace execution path、map architecture layers、列出关键文件清单。

Phase 3: Clarifying Questions。 理解代码库之后，把还不明确的地方一次性列出来问你：edge cases、error handling、integration points、backward compatibility、performance expectations。重点是在开始设计前，先尽量把歧义收敛掉。

Phase 4: Architecture Design。 再派 2~3 个 code-architect agents，从不同视角给出实现思路（最小改动、更干净的架构、pragmatic balance），对比 trade-off，再给 recommendation。

Phase 5: Implementation。 这里才开始真正写代码。命令实现里写得非常明确：没有用户批准，不要开始实现。

Phase 6: Quality Review。 实现完以后，再派 3 个 code-reviewer agents 并行从不同角度看（simplicity/DRY/elegance、bugs/correctness、conventions/abstractions），汇总结果后问你：现在修、以后修、还是就这样继续。

Phase 7: Summary。 最后收尾：标记 todo 完成、总结做了什么、记录关键决策、列出改动文件、给出 next steps。

工作原理

Feature Dev 的实现是三类 agent 组合的 orchestrated command。

code-explorer 负责深挖现有代码库：找 entry points、trace code flow、map architecture layers、找 dependencies 和 side effects、列出必须阅读的关键文件。

code-architect 负责基于现有模式做架构设计：分析现有 conventions、做 architecture decision、明确组件设计、给 phased build sequence。

code-reviewer 负责实现后看质量：看项目规范、看 bug、看代码质量，只报高置信度问题。

这三个 agent 组合起来就是：先理解，再设计，再复查。

它最核心的价值其实不是”更会写代码”，而是”更晚开始写代码”。 普通 Claude Code 最常见的问题，是它太容易在信息不全的时候就开始动手。Feature Dev 用 7-phase workflow 在强行压制这种冲动：需求不清先问、代码库没懂先探索、设计没定先比方案、用户没批准先别实现。

适合谁

适合：要做的 feature 比较大、改动会涉及多个文件或模块、你不想让 Claude 一上来就乱改、你希望它先做 codebase study、你认可”先设计后实现”的节奏。

不太适合：单行小修、简单 bug fix、你已经把要求说得极度清楚只想赶紧改完、你不想被问 clarifying questions。

对于复杂功能，它很有价值；对于简单任务，它会显得有点重。
