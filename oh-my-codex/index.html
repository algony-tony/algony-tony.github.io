如果说 Codex CLI 本体已经是一个很强的终端 agent，那么 oh-my-codex（OMX） 做的事情，就是在 Codex 外面再包一层更完整的工作流和运行时。

它不是另一个模型，也不是一个单独的聊天界面，而是作者自己在 README 里定义的：“a workflow layer for OpenAI Codex CLI”。

项目地址：Yeachan-Heo/oh-my-codex

截至 2026-04-04，我打开仓库页时看到：

  • 仓库 star 大约 15k；
  • 最新 release 是 v0.11.12，发布日期 2026-04-02；
  • README 里主打的一句话是：Start Codex stronger, then let OMX add better prompts, workflows, and runtime help when the work grows.

OMX 是什么

官方 README：oh-my-codex

README 里写得很直白：OMX 不是替代 Codex，而是保留 Codex 作为执行引擎，再额外补上这些东西：

  • 更强的默认启动方式；
  • 一套从澄清需求到完成交付的统一工作流；
  • 一批可复用的技能入口，比如 $deep-interview、$ralplan、$team、$ralph；
  • 把项目 guidance、plan、log、memory、state 固化到 .omx/ 目录。

所以它最像的不是“给 Codex 多一个命令”，而是：给 Codex 增加一个工程化操作系统外壳。

它是不是类似 Superpowers

短答案：是，方向上很像；但 OMX 的范围更大。

我觉得可以这么理解。

相同点

OMX 和我前面写过的 Claude Code 插件 Superpowers，核心思想是一样的：

  • 都不是换模型；
  • 都不是强调 UI；
  • 都是在给 coding agent 加流程、加纪律、加方法论；
  • 都希望把“需求澄清 → 计划 → 实现 → 验证”这套节奏固定下来。

换句话说，二者都属于“让 agent 别一上来瞎写代码”这一派。

不同点

但 OMX 明显比 Superpowers 更重一些。

维度 Superpowers OMX
依附对象 Claude Code Codex CLI
主要形态 Claude 插件 / skills / hooks 独立 workflow layer + runtime
核心目标 给 Claude Code 加方法论 给 Codex 加方法论 + 编排能力 + 状态管理
持久化 主要靠 skill / plugin 约束 明确把计划、日志、memory、state 放进 .omx/
多 agent 协作 有，但更偏 Claude 自身能力 明确有 $team、tmux runtime、团队编排
运行时辅助 相对轻 还有 HUD、explore、sparkshell、doctor 等操作面

所以如果非要打个比方：

  • Superpowers 更像是“Claude Code 的流程强化插件”；
  • OMX 更像是“Codex 的工程化工作流发行版”。

  • Node.js 20+
  • 已安装 Codex CLI
  • 已完成 Codex 认证
  • 如果你要用 team runtime，在 macOS / Linux 上还需要 tmux

也就是说，OMX 不是替代 Codex 的安装包，而是建在 Codex 之上的一层增强。

推荐工作流

官方 README：oh-my-codex

OMX 最重要的不是具体命令，而是它推荐的默认节奏。

README 里给出的 canonical flow 基本就是四步：

这四个入口分别对应：

  1. $deep-interview：先澄清范围、边界、非目标；
  2. $ralplan：把澄清后的需求压成可批准的方案；
  3. $ralph：单 owner 持续推进直到完成并验证；
  4. $team：如果任务适合并行，就拉多 agent 协作执行。

这个设计思路非常明显：先把“想做什么”讲清楚，再决定“一个 agent 做”还是“多个 agent 并行做”。

OMX 增加了什么

1. 统一工作流关键词

对最终用户来说，最容易记住的其实不是具体目录结构，而是这些会话内 surface：

  • $deep-interview
  • $ralplan
  • $ralph
  • $team
  • /skills

这和 Superpowers 有一点像：都是用一套清晰的工作流名词去约束 agent 行为。

2. .omx/ 持久化状态

README 里明确写了，OMX 会把这些东西放进 .omx/：

  • plans
  • logs
  • memory
  • runtime state

这件事非常重要，因为很多 agent 工具的问题不是“不会写代码”，而是：

  • 过程没有持久化；
  • 计划和执行割裂；
  • 一次会话结束后就忘了。

OMX 是明显在补这一层。

4. 操作面工具

README 里还列了几类比较 operator-oriented 的命令：

  • omx setup
  • omx doctor
  • omx hud --watch
  • omx explore --prompt "..."
  • omx sparkshell ...

这就更像一整套工作环境了，而不只是 prompt 包。

它的简单心智模型

我觉得 README 里那句总结非常好：

Codex 做实际 agent 工作，OMX 负责更好的 task routing、更好的 workflow、更好的 runtime。

把它翻成更口语的话就是：

  • Codex 负责“干活”；
  • OMX 负责“让它按工程化流程干活”。

所以我自己会把 OMX 看成下面这几样东西的组合：

  1. 流程模板：什么时候先澄清，什么时候先计划；
  2. 角色系统：不同任务交给不同 agent surface；
  3. 状态系统：计划、日志、memory、mode 都落盘；
  4. 协作运行时：需要时可以拉 team 做并行执行。

适合谁

README 里有一句我很赞同：

如果你只是想要 plain Codex、不要额外 workflow layer，那你可能并不需要 OMX。

也就是说，OMX 最适合的是这类人：

  • 你已经在用 Codex；
  • 你觉得单纯聊天式 agent 不够稳定；
  • 你希望任务更像正式软件开发流程；
  • 你想把多 agent 协作、计划、验证这些东西固定下来。

如果你只是想“偶尔让 agent 改个小文件”，那纯 Codex 可能已经够了。
